Safe vs Idempotent ‚Äî Critical Distinctions

Safe Method

A method is safe if calling it does not modify server state. Safe methods can be cached, prefetched, and crawled.

Safe methods: GET, HEAD, OPTIONS, TRACE

Idempotent Method

A method is idempotent if calling it multiple times produces the same result as calling it once.

Idempotent: GET, HEAD, PUT, DELETE, OPTIONS

üéØ Real-World Analogy

Safe: Looking at a light switch doesn't change whether it's on or off.

Idempotent: Pressing "ON" multiple times still results in the light being on.

Neither: Adding ¬£10 to your bank account is not idempotent ‚Äî doing it twice gives you ¬£20.

‚ö†Ô∏è Why Idempotency Matters

Network failures happen. For idempotent methods, it's safe to retry. For POST, retrying might create duplicate records ‚Äî leading to double charges or data corruption.

CORS ‚Äî Cross-Origin ResursƒÉ Sharing

"It works in Postman but not in the browser" ‚Äî this is almost always CORS.

Same-Origin Policy

Browsers block JavaScript from making requests to a different origin (protocol + domain + port). This prevents malicious websites from stealing data from your sessions.

The Problem

// Page loaded from https://myapp.com fetch('https://api.other.com/data') .then(r => r.json()) .catch(err => console.error('CORS!')); // Blocked unless api.other.com allows it

The Solution

# Server response headers Access-Control-Allow-Origin: https://myapp.com Access-Control-Allow-Methods: GET, POST, PUT Access-Control-Allow-Antete: Content-Type
‚ö†Ô∏è Preflight Cereres

For "non-simple" requests (custom headers, methods other than GET/POST), the browser first sends an OPTIONS request. Your server must handle this.

Evolu»õia HTTP

From HTTP/1.0's connection-per-request to HTTP/3's QUIC ‚Äî how the protocol adapted.

HTTP/1.1 ‚Äî Limitations That Shaped the Future

HTTP/1.1 served us for 15 years, but its design shows its 1997 origins.

üêå Head-of-Line Blocking
Cereres must complete in order. A slow response blocks all subsequent requests on that connection.
üìù Text Antete
Antete are sent as uncompressed ASCII text, repeated with every request. ~500-800 bytes overhead per request.
üîÑ No Server Push
Server can only respond to requests ‚Äî cannot proactively send resources it knows the client will need.
üîó Connection Overhead
Even with keep-alive, each connection has TCP + TLS overhead. Browsers open 6 parallel connections per domain.

HTTP/2 ‚Äî The Multiplexing Revolution

Standardised in 2015, HTTP/2 addressed HTTP/1.1's performance bottlenecks while maintaining backwards-compatible semantics.

Multiplexing
Multiple requests/responses share a single TCP connection as independent "streams". No HTTP-layer head-of-line blocking.
Header Compression (HPACK)
Antete are compressed using static and dynamic tables. Repeated headers aren't resent.
Server Push
Server can proactively send resources before client requests them.

HTTP/3 ‚Äî Built on QUIC (UDP)

HTTP/3 replaces TCP with QUIC ‚Äî solving TCP's head-of-line blocking and enabling 0-RTT connections.

QUIC (Quick UDP Internet Connections)

A transport protocol providing reliable, encrypted, multiplexed connections over UDP, with TLS 1.3 built-in.

‚úì No TCP Head-of-Line Blocking
QUIC streams are independent ‚Äî a lost packet only affects one stream, not all.
‚úì 0-RTT Connection Resumption
Returning clients can send data immediately without waiting for handshake.
‚úì Connection Migration
Connections survive IP address changes (e.g., Wi-Fi to mobile).
‚úì Always Encrypted
TLS 1.3 is mandatory and integrated. Even metadata is encrypted.

WebSocket ‚Äî When HTTP Isn't Enough

HTTP's request-response model doesn't fit real-time bidirectional communication.

Use Cases

  • Live chat ‚Äî Messages push instantly

REST ‚Äî Representational State Transfer

üìö Origin Story

REST was defined by Roy Fielding in his 2000 PhD dissertation. Fielding was a principal architect of HTTP/1.1 and co-founder of Apache HTTP Server.

REST is not a standard ‚Äî it's a set of constraints that yield scalability, simplicity, and evolvability.

The Six REST Constraints
  1. Client-Server ‚Äî Separation of concerns
  2. FƒÉrƒÉ stare ‚Äî Each request contains all needed context
  3. Cacheabil ‚Äî RƒÉspunss must declare cacheability
  4. Interfa»õƒÉ UniformƒÉ ‚Äî Standardised interactions (key constraint)
  5. Layered System ‚Äî Intermediaries transparent
  6. Code on Demand (optional) ‚Äî Server can send executable code
üéØ The Uncomfortable Truth

Most "API RESTs" are actually RPC-over-HTTP. True REST requires hypermedia (HATEOAS) ‚Äî responses include links telling clients what actions are available.

Richardson Maturity Model

Leonard Richardson proposed a model for classifying API designs by how much they embrace REST.

Level 0 & 1 ‚Äî The Anti-Patterns

Level 0: RPC over HTTP

POST /api HTTP/1.1 { "action": "getUser", "userId": 42 }
üö´ Problems
  • Can't cache (everything is POST)
  • Can't retry safely
  • URLs meaningless for debugging

‚úì Progress
ResursƒÉs have identities. /users/42 always refers to the same user.
‚úó Still Wrong

This is where most "API RESTs" live ‚Äî using HTTP methods correctly.

üí° Benefits of Level 2

GET requests can be cached. PUT and DELETE can be retried safely. Proxies understand semantics. Standard HTTP tooling just works.

SOAP is an XML-based messaging protocol for enterprise interoperability with formal contracts.

curl ‚Äî The Swiss Army Knife

curl is the most versatile command-line HTTP client. Master it to test any API.

Essential Flags
-v verbose ‚Ä¢ -X method ‚Ä¢ -H header ‚Ä¢ -d data ‚Ä¢ -L follow redirects ‚Ä¢ -k insecure
Pro Sfat
Browser DevTools can "Copy as cURL" ‚Äî right-click any request in the Network tab.

Common HTTP Issues ‚Äî Quick Diagnosis

"Connection refused"

Cause: Server not running or wrong port

Check: netstat -tlnp | grep PORT

"Connection timed out"

Cause: Firewall blocking, wrong IP

Check: ping host, telnet host port

"SSL certificate problem"

Cause: Self-signed, expired, or wrong hostname

Check: openssl s_client -connect host:443

"CORS error"

Cause: Server not sending CORS headers

Check: Server must send Access-Control-Allow-Origin

405 Method Not Allowed

Cause: Using wrong HTTP method

Check: API documentation

415 Unsupported Media Type

Cause: Missing Content-Type header

Check: Add Content-Type: application/json
