üåê Ce este Internet of Things?

üìñ Formal Defini»õie

The Internet of Things (IoT) refers to a network of physical devices equipped with sensors, software, and connectivity that enable them to collect, exchange, and act upon data without requiring human intervention.

üèóÔ∏è IoT Reference Architecture

üîç Security Boundaries

Each layer transition is a potential attack surface. The Communication and Gateway layers are particularly critical ‚Äî they often translate between constrained protocols (Zigbee, BLE) and IP-based networks.

üì® MQTT ‚Äî The Dominant IoT Protocol

üìñ Message Queuing Telemetry Transport

A lightweight publish-subscribe messaging protocol designed for constrained devices and unreliable networks. Developed by IBM in 1999 for satellite links, now the de facto IoT standard.

Protocol Characteristics

  • Minimal overhead: 2-byte minimum header
  • Bidirectional: Publish and subscribe on same connection
  • Connection-oriented: TCP with keep-alive
  • Three QoS levels: 0, 1, 2

Standard Ports

UNENCRYPTED

Port 1883

Plaintext MQTT ‚Äî credentials and data visible to attackers

TLS ENCRYPTED

Port 8883

MQTT over TLS ‚Äî encrypted end-to-end

‚ö†Ô∏è 2018 Avast Research

~49,000 exposed MQTT brokers on the Internet; 32,000+ had no password protection.

üîÑ MQTT Publish-Subscribe Architecture

Publisher

Sends messages to a topic. Decoupled ‚Äî doesn't know about subscribers.

Broker

Routes messages, handles authentication, manages sessions and QoS.

Subscriber

Registers interest in topics (supports wildcards). Receives matching messages.

üÉè MQTT Topic Wildcards ‚Äî Power and Danger

Wildcard Types

+ Single-Level Wildcard

Matches exactly one topic level.

home/+/temperature # Matches: home/kitchen/temperature # Matches: home/bedroom/temperature # NOT: home/kitchen/sensor/temperature

# Multi-Level Wildcard

Matches any number of levels. Must be last character.

home/# # Matches: home/kitchen/temperature # Matches: home/bedroom/light/status # Matches: home (just the root)

‚ö†Ô∏è Security Implications

Subscribing to # alone receives ALL messages on an unprotected broker. An attacker can:

  • Monitor all device communications
  • Capture credentials in payloads
  • Map the entire IoT topology
  • Identify high-value targets

üõ°Ô∏è Bune Practici

  • Never allow # subscriptions in production
  • Use ACLs to restrict wildcard scope
  • Consistent lowercase naming
  • Embed unique device IDs in topics

üìå MQTT Mesaje Re»õinute ‚Äî Persistent State Risk

üìñ What Are Mesaje Re»õinute?

When a message is published with the retain flag set, the broker stores it. Any new subscriber to that topic immediately receives the last retained message ‚Äî even if it was published hours or days ago.

# Publishing with retain flag client.publish( topic="home/thermostat/status", payload="online", qos=1, retain=True # Broker stores this ) # New subscribers get this immediately

‚ö†Ô∏è Security Risks

  • Information leakage: New subscribers see historical state
  • Stale data attacks: Old messages without timestamps can mislead
  • Message injection: Atacers can publish retained malicious content

üìä Persistence Duration

AWS IoT Core stores retained messages for up to 3 years. Messages persist until explicitly deleted (publish empty payload with retain=true).

üõ°Ô∏è MQTT 5.0 Mitigation

Message Expiry Interval allows automatic expiration of retained messages.

üíÄ MQTT Ultima Voin»õƒÉ and Testament (LWT)

üìñ What is LWT?

Clients can specify a "last will" message during connection. If the client disconnects ungracefully (network failure, crash, timeout), the broker automatically publishes this message.

Legitimate Use Cases

Device Health Monitoring

Instant notification when a sensor goes offline.

Graceful Degradation

Trigger backup systems when primary fails.

‚ö†Ô∏è Atac Vector

Malicious clients can register harmful LWT messages broadcast to all subscribers upon disconnection. Mitigation: ACLs restricting which topics clients can set as Will Topics.

IoT Protocol Landscape

Beyond MQTT: CoAP, Zigbee, Z-Wave security comparison

üì° CoAP ‚Äî The RESTful IoT Protocol

üìñ Constrained Application Protocol (RFC 7252)

A UDP-based protocol designed for constrained devices. Uses HTTP-like methods (GET, POST, PUT, DELETE) but with minimal overhead ‚Äî 4-byte minimum header versus HTTP's 26+ bytes.

Message Types

CON (Confirmable)

Reliable delivery with retransmission and ACK.

NON (Non-confirmable)

Fire-and-forget, no acknowledgement required.

Security Modes

ModeSecurityUse Case
NoSecNoneTesting only
PreSharedKeyDTLS + PSKKnown devices
RawPublicKeyDTLS + RPKConstrained PKI
CertificateDTLS + X.509Full PKI

‚ö†Ô∏è DTLS Overhead

DTLS handshake adds significant overhead for constrained devices. OSCORE (RFC 8613) provides application-layer security as an alternative.

üè† Smart Home

MQTT preferred ‚Äî event-driven (motion detected, door opened), many listeners.

üè≠ Industrial Senzors

CoAP preferred ‚Äî request-response for reading sensor values, low power.

üîã Battery Devices

CoAP preferred ‚Äî UDP allows sleep modes without maintaining TCP connection.

üì∂ Zigbee Security Architecture

üìñ Zigbee Prezentare GeneralƒÉ

IEEE 802.15.4-based mesh networking protocol. Uses 128-bit AES encryption with two security models: Centralised (Trust Center) and Distributed.

Key Types

Network Key

Encrypts broadcast traffic. Shared by all devices in network.

Link Key

Encrypts unicast traffic between device pairs.

‚ö†Ô∏è Default Link Key Atac

Zigbee Home Automation used a well-known default key: "ZigbeeAlliance09". Atacers could decrypt network key transfers during device pairing.

üì° Touchlink Vulnerabilitate (WiSec 2017)

Researchers demonstrated device hijacking from 190 metres using a leaked ZLL Master Key. Atacer could factory reset and steal devices.

‚úì Zigbee 3.0 Improvements

Mandatory Install Codes generate unique per-device link keys. No more default keys.

üîê Z-Wave Security: S0 vs S2

S0 Security (Pre-2017) ‚Äî Critically Flawed

‚ö†Ô∏è All-Zeros Key Exchange

S0 performed network key exchange using an all-zeros temporary key. Any attacker in RF range during pairing could capture the network key.

Temp Key: 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 # Encrypted network key easily decrypted

S2 Security (April 2017+) ‚Äî Mandatory

‚úì ECDH Key Exchange

S2 uses Elliptic Curve Diffie-Hellman with Device-Specific Keys (DSK) for authentication. Three security classes:

  • S2 Access Control: Door locks (requires DSK)
  • S2 Authenticated: Senzors (requires PIN)
  • S2 Unauthenticated: Non-critical devices

‚ö†Ô∏è Z-Shave Downgrade (2018)

Atacers could spoof Node Info to remove S2 capability, forcing fallback to vulnerable S0 pairing. 100+ million devices potentially affected.

üè∞ Defense in Depth ‚Äî Layered Security

üìñ The Principle

Defense in Depth employs multiple layers of protection. If one layer fails, others remain. No single point of failure should compromise the entire system.

A castle doesn't rely solely on its outer wall. It has a moat, drawbridge, inner bailey, keep, and guards at each layer. The fish tank hack succeeded because the casino had only a "single wall" ‚Äî flat network architecture.

üîí Hardware Security Foundations

üìñ Secure Boot Chain of Trust

Each boot stage cryptographically verifies the next before execution. Starts from immutable Hardware Root of Trust (ROM/eFuse).

Hardware Security Modules

Secure key storage, crypto acceleration, random number generation. Keys never leave in plaintext.

Hardware-enforced isolation: Normal World (Rich OS) vs Secure World (TEE) on single CPU.

Production Requirements

Azure IoT Edge and AWS IoT Greengrass require TPM/HSM. Matter smart home standard mandates device attestation.

üîç Port Scanning ‚Äî Theory

üìñ What is Port Scanning?

Systematic probing of a target's network ports to identify listening services and potential vulnerabilities.

Port States

OPEN

Service actively listening and accepting connections.

CLOSED

Port accessible but no service listening. Host responds with RST.

FILTERED

Firewall drops packets silently. No response = filtering.

Scanning Techniques

TechniqueMethodDetection
TCP ConnectFull 3-way handshakeHigh (logged)
SYN ScanHalf-open (stealth)Medium
UDP ScanICMP unreachableLow
FIN/XMAS/NULLUnusual flagsVery low

Port scanning without authorisation may be illegal. Use only in controlled lab environments or with written permission.
