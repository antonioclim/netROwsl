üìú Historical Parallel

In 1854, a British banker needed to send an urgent message from London to Edinburgh. He had two options: send a registered letter (guaranteed delivery, signature required, but slow) or send a telegram (fast, but no confirmation).

Today, every application on your computer faces the exact same choice:

üì® TCP (The Registered Letter)

Reliable, ordered, confirmed delivery. Used when every byte matters.

‚ö° UDP (The Telegram)

Fast, lightweight, no guarantees. Used when speed beats perfection.

This lecture: Understanding how the transport layer delivers data from one process to another across the Internet.

üìπ Video Streaming

üîê Security

üîç Debugging

‚òÅÔ∏è Microservices

Section 1

Rolul Nivelului Transport

Position in the stack, multiplexing, and the socket abstraction

üìö Pozi»õia √Æn Stiva TCP/IP

üìñ Defini»õie

The transport layer provides process-to-process communication, extending the network layer's host-to-host delivery.

Key Responsibilities

  1. Process identification via ports
  2. Multiplexare/demultiplexing
  3. Reliability (TCP only)
  4. Control al fluxului
  5. Control al congestiei

üîÄ Problema MultiplexƒÉrii

üìÆ The Postal Analogy

IP addresses are like street addresses‚Äîthey identify a building. But a building has many apartments (processes). How does the postman know which apartment gets which letter?

Ports are the apartment numbers. Each application "listens" on a specific port.

üî¢ Mecanismul Porturilor

üìñ What is a Port?

A port is a 16-bit unsigned integer (0‚Äì65535) that identifies a specific process or service endpoint on a host.

Section 2

TCP: Calul de BƒÉtaie Fiabil

Stabilire conexiune, reliability, flow control

üìã TCP: Caracteristici Fundamentale

üìñ Transmission Control Protocol (RFC 793, 1981)

TCP is a connection-oriented, reliable, byte-stream protocol that provides ordered, error-checked delivery.

üîó Connection-Oriented

Must establish connection (handshake) before data transfer.

‚úÖ Reliable Delivery

Guarantees all data arrives, in order, without corruption.

üìä Byte Stream

No message boundaries‚Äîcontinuous stream of octe»õi.

‚ÜîÔ∏è Full-Duplex

Simultaneous bidirectional communication.

üì¶ Structura Antetului TCP (20+ octe»õi)

Minimum header: 20 octe»õi. With options: up to 60 octe»õi.

üö© Flag-uri TCP: Semnalele de Control

Flag Name Purpose When Used
SYN Synchronise Initiate connection Stabilire conexiune
ACK Acknowledge Confirm receipt Almost every packet
FIN Finish Gracefully close Terminare conexiune
RST Reset Abort connection Errors, port not listening
PSH Push Deliver immediately Interactive traffic

üí° Packet Analysis Tip

In tcpdump: [S] = SYN, [.] = ACK, [P.] = PSH+ACK, [F.] = FIN+ACK, [R] = RST

ü§ù Handshake-ul √Æn Trei Pa»ôi

One RTT (Timp Dus-√éntors) before data can flow. This is why TCP connections have latency overhead.

‚ùì De Ce Trei Pa»ôi? (Nu Doi?)

Purpose 1

ISN Synchronisation

Both parties exchange Initial NumƒÉr de Secven»õƒÉs.

Purpose 2

Reachability Confirmation

Each side proves it can both send and receive.

Purpose 3

Old Packet Prevention

Delayed packets from old connections rejected.

‚ö†Ô∏è SYN Flood Attack

Attackers send thousands of SYNs without completing handshake. Server resources exhausted.

Mitigation: SYN cookies (RFC 4987)

ü§î What would happen with only 2-way handshake?
Nothing‚Äîit would work fine
Old delayed SYNs could create zombie connections
Connections would be faster
‚úì Correct! Without the third step, servers can't distinguish fresh from delayed requests.

üëã TCP Connection Termination (Four-Way)

Why Four Steps?

TCP is full-duplex‚Äîeach direction closes independently.

‚è±Ô∏è TIME_WAIT State

After final ACK, client waits 2√óMSL (typically 2 minutes) to ensure delayed packets don't confuse future connections.

üí° SO_REUSEADDR

In Python: setsockopt(SOL_SOCKET, SO_REUSEADDR, 1) allows binding to ports in TIME_WAIT.

‚öôÔ∏è TCP Options: MSS (Dimensiune MaximƒÉ Segment)

üìñ Defini»õie

MSS is the largest amount of data TCP will send in a single segment. Negotiated during the handshake.

The Mathematics

MSS = MTU ‚àí IP Header ‚àí TCP Header

For Ethernet (MTU=1500):
MSS = 1500 ‚àí 20 ‚àí 20 = 1460 octe»õi

‚öôÔ∏è TCP Options: SACK (Selective Acknowledgment)

The Problem with Cumulative ACKs

Classic TCP only acknowledges "received up to byte X". If packet 3 is lost, packets 4 and 5 must be retransmitted too!

‚ùå Without SACK

Packets: 1‚úì 2‚úì 3‚úó 4‚úì 5‚úì

ACK stuck at 3. Retransmit 3, 4, 5.

‚úÖ With SACK

ACK 3, SACK 4-5

"I got up to 2, and also 4 and 5. Just resend 3!"

üìñ Benefits

  • Reduces unnecessary retransmissions
  • Faster recovery from multiple losses
  • Essential for high-latency networks

‚öôÔ∏è TCP Options: Window Scaling

üìñ The Problem

TCP header's Window field is 16 bi»õi ‚Üí max 65,535 octe»õi. For high-speed links, this limits throughput!

Effective Window: 8,388,608 octe»õi

Max Throughput: 671.09 Mbps

Effective Window = Window √ó 2scale

Scale=7 ‚Üí 65535 √ó 128 = 8.3 MB window!

Section 3

UDP: The Speedy Minimalist

When reliability takes a back seat to speed

üìã UDP: Fundamental Characteristics

üìñ User DatagramƒÉ Protocol (RFC 768, 1980)

UDP is connectionless, unreliable, datagram-oriented‚Äîessentially just port multiplexing and checksums.

üîì FƒÉrƒÉ conexiune

No handshake, no state. Just send whenever.

‚ùå Unreliable

No delivery guarantees, no ordering. "Fire and forget."

üì¨ DatagramƒÉ-Oriented

Preserves message boundaries. Each send() ‚Üí one recv().

‚ö° Minimal Overhead

Just 8 octe»õi of header. Zero setup latency.

üì¶ UDP Header: Beautiful Simplicity (8 octe»õi)

üéØ When to Use UDP?

üåê DNS

Short queries, fast responses. Single datagram sufficient.

üì° DHCP

Client has no IP yet‚Äîcan't establish TCP! Broadcast needed.

üìπ Video Streaming

Missing frames less harmful than pausing. Real-time matters.

üéÆ Online Gaming

Current position makes previous obsolete. Old data worthless.

üìû VoIP

Real-time audio. 100ms delay worse than occasional click.

üìä Telemetry

High volume, acceptable loss. Best-effort is fine.

üí° The Pattern

Use UDP when: latency matters more than reliability, data becomes stale quickly, or you need broadcast/multicast.

Section 4

TLS: Securizarea Transportului

Confidentiality, integrity, and authentication

üîê Why Do We Need TLS?

‚ö†Ô∏è The Problem

TCP and UDP transmit data in plain text. Anyone on the network path can read your traffic.

Without TLS

With TLS

TLS Provides Three Guarantees

  1. Confidentiality: Data encrypted
  2. Integrity: Tampering detected
  3. Authentication: Server identity verified

üìö TLS: Position in the Stack

üìñ HTTPS = HTTP + TLS

When you visit https:// sites, your browser speaks HTTP inside an encrypted TLS tunnel.

Common TLS-Secured Protocols

  • HTTPS (port 443)
  • SMTPS (port 465)
  • IMAPS (port 993)
  • FTPS (port 990)

ü§ù TLS 1.3 Handshake

TLS 1.3: Reduced to 1 RTT (from 2 in TLS 1.2). Supports 0-RTT for resumed connections!

Section 5

QUIC: Viitorul Transportului

TCP + TLS reinvented over UDP

‚ùì Why Do We Need QUIC?

TCP's Limitations

‚ö†Ô∏è Head-of-Line Blocking

One lost packet blocks ALL data behind it‚Äîeven for different HTTP requests.

‚ö†Ô∏è Slow Setup

TCP handshake + TLS handshake = 2-3 RTT before first byte.

QUIC's Solutions

‚úÖ Independent Streams

Loss on one stream doesn't block others.

‚úÖ Combined Handshake

Transport + crypto in 1 RTT. 0-RTT for resumed.

üèóÔ∏è QUIC Architecture

Traditional Stack

QUIC Stack

üìñ HTTP/3 = HTTP over QUIC

Major sites (Google, Facebook, Cloudflare) already serve significant traffic via HTTP/3.

‚öîÔ∏è QUIC vs TCP+TLS: Comparison

Aspect TCP + TLS QUIC
Handshake 2-3 RTT 1 RTT (0-RTT possible)
Multiplexare HOL blocking Independent streams
Header Encryption TCP header in clear Almost entirely encrypted
Connection Migration IP change = reconnect Connection ID survives

üì± Connection Migration

Phone switches Wi-Fi ‚Üí 4G. TCP: every connection breaks. QUIC: connections continue seamlessly!
